> Pegá este archivo completo al inicio de futuras conversaciones para dar contexto rápido del estado de la migración. Mejor aún: decile a Claude que clone `https://github.com/cristobalroman95/queseproduce.git` para que lea el código directo del repo sin gastar contexto pegando `index.html`.

## Arquitectura objetivo
- **Frontend:** `index.html` (single-file) servido por Cloudflare Pages, deploy automático desde GitHub.
- **Repo:** `cristobalroman95/queseproduce`, branch `main`.
- **Backend:** Supabase (Postgres + Auth). Conexión directa navegador → Supabase vía `supabase-js`. No hay Cloudflare Functions/Workers todavía.
- **Seguridad:** depende 100% de RLS, porque la `anon key` queda pública en el código fuente del HTML.

## Archivos clave del repo
- `index.html` — versión en producción (la que sirve Cloudflare Pages). Tiene el cliente Supabase, login y la tabla `shows` migrados.
- `QueseProduce_2026_rediseño_4_3.html` — versión vieja, pre-Supabase (login con PIN). **No se usa en el deploy.** Candidata a borrar o archivar.
- Cliente Supabase: línea ~1016-1018 de `index.html` (`SUPABASE_URL`, `SUPABASE_ANON_KEY`, `const sb = supabase.createClient(...)`).
- `loadShows()`, `saveShow()`, `saveShows()` — funciones núcleo de carga/guardado de shows y sus relacionadas.

## Patrón de código a seguir en cada paso (validado con `shows`, `ficha_tecnica`, `roadmap` y `presupuesto_items`)
- Reemplazar la función `load*()`/generación local (`default*()`, `fresh*()`) que llena el dato por una versión `async` con `sb.from(tabla).select("*")`, filtrando/agrupando por `show_id` cuando corresponda.
- **Datos 1:1 simples sin sub-listas** (ej. `ficha_tecnica`): `upsert(payload, {onConflict:"show_id"})` en cada edición de campo. Más liviano, no hay que borrar nada.
- **Datos 1:N o anidados donde se agregan/eliminan ítems** (ej. `roadmap`, `presupuesto_items`, probablemente `cierre_items`): estrategia de borrar todas las filas del show y reinsertar desde el estado en memoria (`persistRoadmap`/`persistPresupuesto` como referencia). Encadenar los guardados con un chain tipo `roadmapSaveChain`/`presupuestoSaveChain` para evitar carreras si hay ediciones rápidas seguidas.
- Antes de insertar sin `id` en una tabla nueva, correr el equivalente a:
  ```sql
  alter table <tabla> alter column id set generated by default;
  ```
  (si la PK es `GENERATED ALWAYS AS IDENTITY`, el insert sin `id` falla).
- Al crear un show nuevo (`saveShows()`), después de asignarle el `id` real, crear también las filas iniciales en las tablas relacionadas ya migradas (como se hace hoy con `ficha_tecnica`, `roadmap` y `presupuesto_items`).
- Errores: mostrar el toast existente (`#toast`) en vez de fallar silenciosamente — ya implementado así en todas las funciones de guardado.
- Decidir si `localStorage` se mantiene como cache de lectura o se elimina del todo (pendiente, sigue sin resolver).
